Architectural Decisions
Frontend-First Development
Decision

Develop the frontend independently before integrating any backend.

Reason

The primary objective is to establish a professional, reusable UI/UX foundation without being blocked by backend implementation.

The frontend should remain backend-agnostic and communicate through APIs later.

shadcn/ui as the Design System Foundation
Decision

Use the official shadcn/ui components as the project's UI foundation.

Reason

The project should build upon a mature and accessible component system rather than recreating common UI primitives.

Application-specific functionality will extend these components instead of replacing them.

Composition Over Modification
Decision

Generated shadcn components should never be modified unless absolutely necessary.

Reason

Application-specific logic should remain separate from generated files to simplify upgrades and maintenance.

Example:

components/ui/sidebar.tsx      в†ђ Generated by shadcn

components/layout/AppSidebar.tsx   в†ђ Project implementation
Build Layout Before Features
Decision

Create the reusable Application Shell before implementing feature pages.

Reason

Every page in the application should inherit a common layout rather than implementing its own structure.

Incremental Project Structure
Decision

Only create folders when they immediately serve a purpose.

Reason

Avoid unnecessary empty folders.

Every folder introduced into the project should immediately contain files with clear responsibilities.

Modern TypeScript Standards
Decision

Use import type for type-only imports.

Example:

import type { ReactNode } from "react";
Reason

The project uses the latest TypeScript configuration with verbatimModuleSyntax, making type-only imports the preferred convention.

Development Rules
One Step at a Time

Development is performed in small, verifiable steps.

Each step must be completed and confirmed before beginning the next.

No skipping ahead.

No combining multiple implementation steps into one response.

Explain Before Coding

Before any implementation, explain:

Goal
Decision
Reason
Files Affected

Verify Before Continuing

Every completed step must be tested.

If a problem occurs:

Stop.
Fix the problem.
Verify the solution.
Continue only after successful verification.
Every File Must Have a Purpose

Before creating a file, explain:

Why it exists.
Why it belongs in its folder.
Whether it is reusable.
Which files depend on it.
Every Folder Must Have a Purpose

Folders should only be created when they are immediately needed.

Avoid speculative or placeholder folder structures.

Review Project Sources First

Before beginning any development discussion:

Review the Project Constitution.
Review previous Project Source Updates.
Respect previous architectural decisions.
Avoid repeating previously completed work.

Communication Rules

The communication workflow has been refined.

Every development response should follow this structure:

Goal

State the immediate objective.

Why

Briefly explain why this step is being performed.

Files Affected

Specify exactly which files will be created or modified.

Do This Now

Provide one and only one implementation task.

Examples:

Create one file.
Modify one file.
Run one command.

Avoid combining multiple independent tasks.

Stop Here

Every response must clearly specify what should be returned.

Examples:

Success
Error message
Screenshot

No additional implementation should occur until verification is complete.

New Permanent Rule
Never Leave the Developer Wondering What Comes Next

Every development response must end with exactly one actionable task.

The developer should never finish reading a response and wonder:

"What am I supposed to do now?"

Every response must conclude with:

One clear implementation task.
One expected result.
One verification request.

If the task succeeds, proceed to the next step.

If it fails, solve that problem before introducing any new work.

Discussion about future architecture or upcoming milestones should never replace the immediate implementation task.

Lessons Learned

During project setup, the following observations were made:

The latest Vite + TypeScript template requires explicit path alias configuration.
The latest shadcn/ui CLI expects those aliases before initialization.
Modern TypeScript favors import type for type-only imports.
Building around generated components is preferable to modifying them directly.
Verifying every installation step before proceeding prevents compounded configuration issues.
